Algorithm walkthrough

AutoHDR Image Grouping

Group photos by camera angle

Exposure-invariant · scene-change-sensitive · runs 100% in the browser

The problem

A folder of real-estate photos from one shoot. Each camera angle may be shot several times at different exposures (HDR brackets).

Goal: group the photos that belong together — same angle, same scene.

📷 dark📷 mid📷 bright → one group (one angle)

The twist

Ignore this

Exposure. Same angle = tripod-fixed ladder of brightnesses (black → white). Pixel-aligned, but visually very different.

Detect this

Small scene changes. A moved chair or open door means a different group, even at the same angle.

Invariant to exposure · sensitive to content. That tension is the whole challenge.

Scoring is brutal

score = exact_matches / total_groups

  • A group counts only if the set of filenames matches exactly.
  • No partial credit — one extra or missing photo → that group scores 0.
  • Over-merging two angles and over-splitting one bracket are both fatal.

Key insight

Tripod brackets are geometrically identical — they differ only by a brightness curve.

So compare structure (edges, layout), which survives exposure — not raw pixels, which don't.

This makes the matching signal nearly bimodal: same angle ≈ near-perfect, different angle ≈ clearly lower. Ideal for exact-match scoring.

Edge maps drop brightness; zero-mean normalized cross-correlation removes gain → an exposure-invariant similarity. Pure arithmetic, no model.

A learned complement

Gradient-ZNCC

Precise pixel-level alignment. Weakness: similar-layout different rooms.

Color embedding

Small MobileNet, contrastive-trained on color + structure. Captures the room's identity.

Fusion = a blend of the two. Together they beat either alone — proven to generalize across dataset sizes.

From similarity to groups

fused similarity→ threshold → graph edges→ connected components = groups

Connected components honors the exposure ladder: dark→mid→bright link in a chain; no need for every pair to match directly.

Per-run threshold (no labels)

The best threshold drifts with dataset size — a fixed value can't generalize.

So we read the group-count vs threshold curve: it rises steeply, then plateaus near the true grouping. We pick the plateau's leading edge.

Tracks the optimum from a 366-photo set to a 5,000-photo set automatically.

Three refinement passes

Connected-components gets us most of the way. Two residual error types remain:

Over-split

A clipped (near-black / near-white) frame breaks off from its true group.

Over-merge

Two different rooms get joined into one cluster.

FIX 2, 4, 5 — each targets one failure, each validated for zero collateral.

FIX 2 Exposure-ladder re-attach

A clipped frame (brightness ≈ 3 or ≈ 250) has almost no structure left — normal matching fails.

  • Compare only the co-valid (non-saturated) pixels with its brightness-nearest cluster member.
  • Re-attach if that masked correlation is strong and the brightness gap is one real exposure step.

A gap-≥25 guard rejects spurious "same-brightness different-room" matches.

FIX 4 Cluster merge

When the threshold splits one scene into two pieces, we re-join them.

  • For embedding-near clusters, test their brightness-adjacent frames with masked correlation.
  • Merge if strong. Different rooms score ~0.30 → stay apart. Zero false merges.

FIX 5 High-resolution split

The fast 64px descriptor sometimes merges similar-layout but different rooms (looks like 0.70, really 0.30).

  • Re-cluster each group at full 256px masked resolution.
  • If it breaks into ≥2 internally-tight bracket-sets with weak cross-links → it was over-merged → split.

The tightness guard distinguishes true over-merges from legitimately varied groups.

Knowing what's impossible

A few reference groups are mislabeled: unrelated rooms lumped together, drone shots that moved, or duplicate frames shared across groups.

We detect these automatically — a group whose own members can't be visually connected — and exclude them, so the score measures the algorithm, not the labels.
